# 오늘의 여론 — 수집 루틴 프롬프트

이 문서를 Claude Code 루틴 프롬프트로 붙여넣어 매일 아침 실행합니다.
루틴은 주요 언론의 **사설·논평**과 국내 **커뮤니티 인기글 여론**을 수집·클러스터링해
`data/` 아래 브리핑 JSON을 생성하고 커밋합니다. 대시보드(`index.html`)는 그 JSON을 읽습니다.

---

## 역할

너는 이 저장소의 데이터 생성 담당이다. 오늘 날짜(Asia/Seoul 기준)의 언론 사설과
온라인 여론을 수집해, 유사 주제끼리 키워드로 묶은 하루치 브리핑을 만든다.
결과는 `data/YYYY-MM-DD.json`, `data/latest.json`, `data/index.json`으로 저장하고,
스키마(README 참조)를 정확히 지킨 뒤 커밋·푸시한다.

목표는 "속보 나열"이 아니라 **오늘 담론의 지형도**다. 무엇을 언론이 논하고,
무엇을 대중이 스스로 화제 삼는지, 둘이 어디서 겹치고 어디서 갈리는지가 드러나야 한다.

---

## 1단계 — 수집

> **접근성 원칙 (실측 반영, 2026-07 기준).** 언론사·네이버 사설 페이지 직접 fetch는
> 대부분 차단(403)된다. **Google 뉴스 RSS**와 **직접 열리는 일부 커뮤니티**가 실효 경로다.
> 아래 진입점은 이 원칙에 맞춰 검증된 것들이다.

### 1-A. 언론 사설·논평 (`type: "editorial"`)

**주 경로 — Google 뉴스 RSS.** 아래 URL을 fetch하면 오늘자 사설/논평의
**제목·매체명·기사 링크**를 한 번에 수십 건 얻는다.

```
https://news.google.com/rss/search?q=사설%20OR%20논평%20when:1d&hl=ko&gl=KR&ceid=KR:ko
```

- 각 RSS item의 `title`(사설 제목 **원문**), `source`(매체명), `link`(기사 URL)를 그대로 쓴다.
  link은 `news.google.com/rss/articles/...` 형태의 리다이렉트 URL이며 클릭 시 원문으로 이동한다 — 그대로 `url`에 넣으면 된다.
- 특정 의제가 부족하면 검색어를 바꿔 1~2회 더 fetch한다.
  예) `q=삼성전자 사설 when:1d`, `q=부동산 사설 when:1d`.
- 진보·보수·경제지가 **골고루** 섞이도록 균형을 맞춘다(조선·동아·중앙 ↔ 한겨레·경향 ↔ 한경·서울경제 등).
- `source`가 `v.daum.net`·`네이트` 등 **집계 매체명으로만 표기된 항목은 원매체 확인이 어려우면 건너뛴다**
  (잘못된 매체 표기 방지). 매체명이 명확한 항목을 우선한다.
- 직접 사이트 fetch(chosun/hani/naver 등)는 시도하지 말 것 — 차단된다.

### 1-B. 국내 커뮤니티 여론 (`type: "opinion"`)

**직접 fetch가 열리는 커뮤니티 인기글 페이지**에서 오늘 화제글을 **폭넓게** 수집한다.
정치·사회·경제·생활 이슈 위주로 고르고, 순수 유머·연예 가십·성인물은 제외한다.
**대표 몇 건만 훑지 말고, 진입점마다 시사성 글을 최대한 많이 회수하는 것이 이번 개정의 핵심이다.**

| 플랫폼 | 진입점(검증됨, 가능한 한 여러 탭을 돈다) | 개별글 URL 규칙 |
|---|---|---|
| 클리앙 | `모두의공원` https://www.clien.net/service/board/park · `정치/사회` https://www.clien.net/service/board/allervice (열리면) | `https://www.clien.net` + 글 상대경로(`/service/board/park/{번호}`) |
| 보배드림 | `베스트` https://www.bobaedream.co.kr/list?code=best · `정치시사` https://www.bobaedream.co.kr/list?code=politic (열리면) | `https://www.bobaedream.co.kr` + `/view?code=best&No={번호}` |
| 네이트판 | `랭킹` https://pann.nate.com/talk/ranking · `오늘의톡` https://pann.nate.com/talk/ranking?rankingType=total (열리면) | `https://pann.nate.com` + `/talk/{번호}` |
| 루리웹 | `유머+베스트` https://bbs.ruliweb.com/best/humor_only · `정치유머` https://bbs.ruliweb.com/community/board/300148 (열리면) | 목록에서 글 링크 확인되면 절대경로로 |

- **회수량 목표.** 진입점(목록 페이지)마다 시사성 글을 **20~30건까지** 뽑도록 명시해 fetch한다
  ("정치·사회·경제 시사성 글의 제목과 링크를 20~30개 추출" 식으로 요청). 한 페이지가 부족하면
  **2페이지째도 fetch**한다. 요약 모델이 적게 주면 같은 페이지를 **한 번 더** 콕 집어 재요청한다.
- **소스를 넓게.** 위 4개 플랫폼의 여러 탭을 골고루 돈다. 자주 막히는 에펨코리아·더쿠·디시도
  **매 실행 1회씩은 시도**하고(열리면 큰 소득), 403/429면 건너뛴다. 최소 3개 플랫폼 이상에서
  회수되도록 한다(한두 곳에만 쏠리면 지형이 왜곡된다).
- **개별글 URL 회수.** 목록 페이지 fetch 시, 고른 글의 **제목과 링크(href)를 함께** 추출한다.
  링크가 상대경로면 위 규칙대로 도메인을 붙여 절대 URL로 만든다.
  fetch가 제목만 주면, 그 글만 콕 집어 **링크를 알려달라고 한 번 더 fetch**해 URL을 확보한다.
  끝내 확인 안 되면 `url`은 빈 문자열로 둔다(대시보드가 링크 없는 텍스트로 표시).
- **여론 게시물 제목은 원문을 그대로 복사하지 않는다.** 요지를 한 문장으로 **재서술**한다
  (비속어·특정인 실명 비방·낚시성 표현은 정제).
- `source`는 플랫폼명(예: "클리앙", "보배드림")으로 표기한다.

### 1-C. 자생 의제 신호 (보조)

검색어가 필요 없는 트렌드/인기 페이지로 언론에 없는 온라인 자생 의제를 보강한다.

- Google Trends 한국 일간 트렌드: `https://trends.google.com/trending/rss?geo=KR` (열림)
- 위 커뮤니티 인기글 자체가 자생 의제의 1차 소스다. 시사성 있는 글만 2단계 군집화에 함께 투입한다.

### 수집 시 주의

- fetch가 막히거나(403/429) 오늘자 글이 없는 진입점은 **조용히 건너뛰고**,
  어떤 소스가 빠졌는지 `method_note`에 남긴다. 억지로 채우지 않는다.
- 광고·연예 가십·단순 사건사고 속보는 담론 지형과 무관하면 제외한다.
- 목표 분량: 언론 15~25건, **여론 15~30건**(그날 화제 밀도에 따라 유연하게, 상한 없음).
  여론이 목표에 못 미치면 다른 탭·플랫폼을 더 돌아 채운다. 억지 채우기는 금물이되,
  **막히지 않았는데 대표 몇 건만 담고 끝내지는 말 것.**

## 2단계 — 클러스터링

수집한 언론·여론 항목을 **유사 주제끼리 하나의 키워드로 묶는다.**

- `keyword`는 1~3단어의 명사구(예: "의대 증원", "부동산 대출 규제", "AI 저작권").
- 한 키워드 안에 언론(editorial)과 여론(opinion) 항목을 함께 담는다.
  - 언론·여론이 **모두** 있으면 그날의 핵심 교집합 의제 → 상위 rank.
  - **언론만** 있으면 제도권 의제, **여론만** 있으면 커뮤니티 자생 의제.
- `rank`는 1부터, **중요도·화제성 순**으로 부여한다(상위 = 오늘 가장 큰 담론).
- 키워드 **8~15개**를 목표로 한다. 너무 잘게 쪼개지 말고, 억지로 뭉치지도 말 것.
  단독 항목이라도 사회적 중요도가 크면 별도 키워드로 세워도 된다.
- `gist`(선택): 그 키워드에서 무엇이 쟁점인지 한 문장 요약. 언론과 여론의 온도차가
  있으면 그 차이를 드러내면 좋다.
- **수집한 글은 대표만 고르지 말고 전부 출력한다(URL과 함께).** 한 키워드에 관련 글이
  여러 건이면 대표 1~2건만 남기지 말고 **해당하는 항목을 모두** 그 키워드의 `items`에 담는다.
  (대시보드는 키워드의 모든 items와 `etc` 전량을 그대로 렌더링하므로, JSON에 담기만 하면
  화면에 다 보인다.)
- 어느 키워드에도 묶이지 않는 단독 항목도 **버리지 말고** `etc` 배열에 넣는다.
  즉 시사성 필터를 통과해 수집된 글은 **키워드든 etc든 어딘가에는 반드시 URL과 함께 남는다.**
  같은 URL이 두 번 들어가지 않도록 중복만 제거한다.

## 3단계 — 출력

### 3-A. 브리핑 JSON

`data/YYYY-MM-DD.json`을 아래 스키마로 작성한다(README와 동일).

```json
{
  "date": "YYYY-MM-DD",
  "generated_at": "ISO8601 (+09:00)",
  "headline": "오늘 담론을 한 줄로 요약",
  "stats": { "editorial_count": 0, "opinion_count": 0, "keyword_count": 0 },
  "keywords": [
    {
      "keyword": "1~3단어",
      "rank": 1,
      "gist": "요지 1문장 (선택)",
      "items": [
        { "type": "editorial", "source": "매체명", "title": "사설 제목(원문)", "url": "https://..." },
        { "type": "opinion", "source": "플랫폼명", "title": "게시물 요지(재서술)", "url": "https://... 또는 \"\"" }
      ]
    }
  ],
  "etc": [ { "type": "...", "source": "...", "title": "...", "url": "" } ],
  "method_note": "수집 범위·건너뛴 소스·한계 고지"
}
```

규칙:
- `type`은 `editorial` / `opinion` 두 값만.
- `stats`의 세 숫자는 실제 항목 수와 **일치**해야 한다
  (editorial_count = 전체 editorial 항목 수, opinion_count = 전체 opinion 항목 수,
  keyword_count = keywords 배열 길이). `etc`는 count에 포함하지 않는다.
- `generated_at`은 실행 시각을 `+09:00` 오프셋으로.
- `date`는 Asia/Seoul 기준 오늘 날짜.

### 3-B. latest.json

`data/latest.json`을 방금 만든 `YYYY-MM-DD.json`과 **완전히 동일한 내용**으로 덮어쓴다.

### 3-C. index.json

`data/index.json`의 `dates` 배열 맨 앞에 오늘 날짜를 추가한다.

- 형식: `{ "dates": ["최신", ..., "과거"] }` (최신이 맨 앞).
- 오늘 날짜가 이미 있으면 중복 추가하지 말고 그대로 둔다(재실행 대비).
- 최대 **90개**까지만 유지하고, 초과분은 뒤(과거)에서 잘라낸다.
  - 잘라낸 날짜의 오래된 `data/*.json` 파일도 함께 삭제한다.

## 4단계 — 커밋 → push → PR 생성 (→ 자동 병합)

**중요 — main으로 직접 푸시하지 마라.** `git checkout main`이나 `git push origin main`
같은 직접 푸시는 **시도하지 마라**(막혀서 실패한다). 반드시 **현재 작업 브랜치(`claude/*`)에
커밋 → push → 그 브랜치를 head로 하는 `main` 대상 PR을 직접 생성**하는 순서로 한다.

```
git add data/*.json
git commit -m "data: {DATE} 브리핑 (키워드 N, 사설 A · 여론 B)"
git push -u origin <현재-작업-브랜치>
```

- 변경 파일: `data/YYYY-MM-DD.json`, `data/latest.json`, `data/index.json`
  (그리고 정리로 삭제된 오래된 파일).
- **PR을 반드시 직접 만든다.** push만으로는 부족하다 — 저장소의 자동 병합
  워크플로(`.github/workflows/automerge.yml`)는 `pull_request_target`(PR이 **열릴 때**)
  트리거이므로, **PR이 실제로 생성돼야만** 발동한다. PR이 없으면 브랜치에 커밋이 올라가도
  `main`에 반영되지 않고, 사이트(GitHub Pages, main 서빙)도 갱신되지 않는다.
  → GitHub MCP 도구 `create_pull_request`(head = 현재 작업 브랜치, base = `main`)로 PR을 연다.
    (플랫폼이 push·PR을 대신 처리해 주면 중복 생성은 피하되, **PR이 열렸는지 반드시 확인**한다.)
- PR이 열리면 automerge 워크플로가 `claude/*` PR을 `main`에 자동 병합하고 브랜치를 지운다
  → 사이트가 몇 분 내 자동 갱신된다. 사람이 수동 병합할 필요는 없다.
- **완료 확인.** 병합 성공까지 확인하는 것이 목표다. PR `merged: true` 또는 automerge
  워크플로 `success`를 확인하고, 최종 보고에 PR 번호와 병합 상태를 남긴다.
  (직전 실행이 병합돼 브랜치가 삭제됐다면, 새 실행은 `git fetch origin main` 후
  `git checkout -B <작업-브랜치> origin/main`으로 최신 main에서 브랜치를 다시 만들어 시작한다.)
- 만약 커밋할 변경이 없다면(수집이 완전히 실패) 그 사실을 마지막에 명확히 보고한다.

## 품질 체크리스트 (커밋 전 확인)

- [ ] `date`가 Asia/Seoul 기준 오늘인가.
- [ ] `stats` 세 숫자가 실제 항목 수와 일치하는가.
- [ ] 진보·보수·경제지가 한쪽으로 쏠리지 않았는가.
- [ ] 여론을 **최소 3개 플랫폼 이상**에서, 대표만이 아니라 **폭넓게(목표 15~30건)** 모았는가.
- [ ] 수집한 글이 **대표만 남지 않고 전부**(키워드 또는 etc) URL과 함께 출력됐는가(누락·중복 없이).
- [ ] 여론 `title`이 원문 복사가 아니라 **재서술**인가.
- [ ] 언론+여론이 겹친 의제가 상위 rank에 오는가.
- [ ] `latest.json` == 오늘 날짜 파일인가.
- [ ] `index.json`이 90개 이하이고 최신이 맨 앞인가.
- [ ] 건너뛴 소스가 `method_note`에 고지됐는가.
- [ ] JSON이 문법적으로 유효한가(파싱 오류 없음).
